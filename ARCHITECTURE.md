# OPS-LAB — архитектура системы агентов

Таск-менеджер Ксении с автосбором из TG-чатов и автосортировкой.
Читать любому агенту при первом касании проекта.

---

## Текущее состояние (v1)

```
TG чаты → Сборщик (Cowork, 4x/день) → inbox в data.json
                                              ↓
                              Сортировщик (Cowork, 2x/день)
                                              ↓
                              fire / tasks / backlog
                                              ↓
                              launchd autopush → GitHub Pages
                                              ↓
                              Ксения видит на телефоне / Mac (OPS LAB)
```

**Ksenia :: inbox** (личный TG-канал) → каждое сообщение = задача в inbox.

---

## Структура data.json

| Секция | Что | Кто обновляет |
|--------|-----|---------------|
| `inbox` | Входящие, ещё не отсортированы | Сборщик (будущее) или агент вручную |
| `fire` | Горящие: сегодня/завтра, макс 5-7 | Ксения через агента |
| `tasks` | Активные задачи по категориям | Ксения через агента |
| `backlog` | Идеи, бэклог, долгоиграющие проекты | Агент при анализе, Ксения |
| `playbook` | OPS Playbook — 16 процессов | Редко, при изменении процессов |
| `goals` | Цели Юли | После синков с Юлей |
| `waiting` | Заблокировано, ждём от кого-то | Агент при обнаружении блокера |
| `metrics` | Ключевые метрики | Сборщик аналитики (будущее) |
| `horizons` | 30/60/90 дней | Раз в 2 недели |

## Уровни карточек

```
inbox → сортировка → fire | tasks | backlog
                              ↓
                            done (done: true в data.json, задача остаётся)
                              ↓
                            log (append-only запись)
```

Done state:
- UI read-only — чекбоксы показывают состояние из data.json, не кликабельны
- Ксения отмечает done через агента: "отметь f3 как done"
- Агент ставит `done: true` + пишет в `log` + push → синхронизировано на всех устройствах
- Задача НЕ удаляется из секции, остаётся видна (перечёркнута, прозрачна)

- **inbox** — только что прилетело, без приоритета. Показывается в табе "горит" сверху оранжевым блоком.
- **fire** — делать сегодня/завтра. Красный блок.
- **tasks** — активная работа этой/следующей недели. По категориям с приоритетами (fire/high/mid/low).
- **backlog** — идеи, someday/maybe, долгоиграющие проекты. По категориям. Не теряются, можно открыть и подумать.
- **playbook** — справочник процессов. Не задачи, а описания "как делать".

## Лог изменений (log)

`data.json.log` — append-only массив. Каждый агент пишет в лог при изменении задач.

```json
{ "date": "2026-03-27", "action": "done", "id": "f3", "title": "онбординг v1" }
{ "date": "2026-03-27", "action": "moved", "id": "i5", "title": "...", "from": "inbox", "to": "tasks" }
{ "date": "2026-03-27", "action": "added", "id": "i8", "title": "..." }
```

Действия: `done`, `added`, `moved`, `removed`.

## Целевая архитектура (v2) — план

### Слой 1: Сборщики (автоматические, дешёвые)

```
[TG чаты] → Сборщик чатов → inbox в data.json
[TG каналы] → Сборщик аналитики → metrics в data.json
```

**Сборщик чатов:**
- Запуск: cron каждые 4 часа (или Claude Desktop scheduled task)
- Модель: haiku (~1-2K токенов за проход)
- Что делает: читает последние сообщения из рабочих чатов через tgmcp-read, извлекает задачи/решения/дедлайны, кладёт в inbox
- MCP: tgmcp-read → tg_get_messages

**Сборщик аналитики:**
- Запуск: cron раз в день
- Модель: haiku (~1K токенов)
- Что делает: дёргает tg_get_participants, tg_get_message_count, считает churn-сигналы, обновляет metrics
- MCP: tgmcp-read → tg_get_participants, tg_get_message_count, tg_get_stats

### Слой 2: Сортировщик (2 раза в день)

- Запуск: утро + вечер (или по запросу Ксении)
- Модель: sonnet (~2-3K токенов)
- Что делает:
  1. Читает inbox
  2. Для каждого элемента решает: fire / tasks (какая категория, приоритет) / backlog / шум
  3. Обновляет data.json
  4. git push
- Плюс: проходит по fire, убирает то что стало неактуально

**Быстрый режим (по запросу):**
Ксения говорит "пройдись по горящим" → агент читает inbox + fire, пересортирует, пушит.

### Слой 3: Ксения + Claude (стратегия)

Любой агент (Desktop, Code, Codex) при старте:
1. Читает data.json (3KB) — мгновенно в контексте
2. Читает CLAUDE.md — знает как обновлять
3. Читает memory — знает контекст проекта

Ксения обсуждает задачи, агент обновляет board, push, готово.

## Будущие расширения

### CV-LAB блок
Отдельная секция в data.json: `cvlab_tasks`, `cvlab_backlog`.
Отдельная вкладка в UI или отдельный board (решить позже).

### Личные проекты
Аналогично — секция `personal` в data.json или отдельный board.

### Уведомления
Агент генерирует текст уведомления → отправляет через tgmcp-actions.
ToV берётся из blr-tone-of-voice.md.

### Обучаемость
- Memory system накапливает feedback между сессиями
- Сортировщик читает memory → со временем точнее определяет приоритеты
- Ксения корректирует → агент сохраняет feedback memory

---

## Принципы

1. **Агенты не принимают решений** — только структурируют и передают
2. **Минимум токенов** — haiku для рутины, opus для стратегии
3. **data.json = единый источник правды** — не Trello, не Notion, не голова
4. **Любой агент может подхватить** — CLAUDE.md + data.json = полный контекст
5. **Система растёт постепенно** — сначала ручной режим, потом автоматизация

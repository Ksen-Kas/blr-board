# Joe v2 MVP — Описание продукта для сравнения с конкурентами

**Версия:** 1.1  
**Дата:** 2026-02-26  
**Назначение:** Короткое и понятное описание MVP + техническая реализация для ориентира команде.

**Git repo:** https://github.com/Ksen-Kas/joe_services

---

## 1. Краткое позиционирование

| Параметр | Описание |
|----------|----------|
| **Название** | Joe / JOE_APPLY |
| **Класс продукта** | Execution agent для откликов на вакансии (не коуч, не стратег, не чат-ассистент) |
| **Суть** | Агент готовит **готовые к отправке** CV и cover letter под каждую вакансию из единого канона резюме, с явным аппрувом человека и трекингом в pipeline |
| **MVP-принцип** | Фундамент для роста: проект эволюционирует, никогда не «конечный» продукт |

## 2. Целевой пользователь и сценарий

### Кто
- **Профиль:** Соискатель (senior/executive), который откликается на много вакансий и хочет сохранять качество и честность материалов.
- **Контекст:** Один канонический CV, одна «история правды» (CLIENT_SPACE), массовые отклики без выдумывания фактов.
- **Ограничение MVP:** Один пользователь на инстанс (single-tenant), данные в Google Sheets.

### Сценарий использования (end-to-end)
1. Пользователь вставляет **пачку ссылок** (LinkedIn/другие) или текст JD.
2. Joe **парсит**, извлекает контакт из JD, делает **go/no-go** (скоринг + флаги).
3. Пользователь смотрит карточку вакансии, решает: **Подготовить** или **Skip**.
4. Joe **адаптирует CV** (акценты, порядок навыков, ключевые слова) и показывает изменения (track-changes style) → пользователь аппрувит.
5. Joe **генерирует cover letter** (150–250 слов, тема + текст), при необходимости — с учётом заметок пользователя → аппрув.
6. Пользователь **скачивает PDF** (CV + письмо) или копирует текст и отправляет отклик вручную.
7. В pipeline фиксируется статус; **напоминания** (stale new, follow-up, no response) отображаются в UI.
   - Примечание: автоматические Telegram‑напоминания пока берут даты из колонок Follow‑up 1/2.

## 3. Ценностное предложение

| Для пользователя | Как это реализовано в продукте |
|------------------|--------------------------------|
| **Не выдумывать факты** | Жёсткое правило No-Invention: все утверждения сверяются с каноном; CV и письма можно защитить на интервью |
| **Один источник правды** | CLIENT_SPACE (canonical resume + strategy + process) — один канон, множество адаптированных откликов |
| **Контроль перед отправкой** | Каждый шаг (скоринг → CV → письмо) с явным аппрувом; ничего не уходит «в автомате» без просмотра |
| **Скорость без потери качества** | Пакетный ввод ссылок, автоматический скоринг и извлечение контактов, готовые тексты под копирование/PDF |
| **Прозрачность решений** | Fit (Strong/Stretch/Mismatch), флаги (visa, citizenship, exp_gap, junior_role, strong_mismatch), объяснение по каждой вакансии |
| **Трекинг без отдельного инструмента** | Pipeline в том же продукте: статусы, напоминания, дубликаты, дашборд |

## 4. Функциональность MVP (что входит в первую версию)

### 4.1 Вход и парсинг
- Ввод: **URL вакансии** или **сырой текст JD** (в т.ч. пачка ссылок).
- Извлечение: компания, роль, локация (если есть), **контакт из текста JD**.
- Проверка дубликатов: по точному URL и fuzzy (Company + Role + Region).

### 4.2 Скоринг и флаги (Go/No-Go)
- **Fit:** Strong / Stretch / Mismatch с кратким объяснением.
- **Флаги:**
  - 🔴 Stop: visa_required (USA + work visa), citizenship (EU/other).
  - 🟡 Warning: exp_gap (показать %), junior_role (≤5 лет).
  - 🔴 Review: strong_mismatch (тип роли/домен) — человек решает.
- Локация/релокация **не** блокируют и не являются флагом.

### 4.3 CV Tailoring
- **Можно менять:** акценты в summary, порядок навыков, акценты в последних двух позициях, ATS-ключевые слова (если есть в каноне и JD).
- **Нельзя:** факты, даты, метрики, названия компаний/должностей, структура опыта, добавление достижений.
- **CANON CHECK** в каждом выводе: OK / WARN / FAIL / BLOCKED + Mismatch/Unverified/Fix.
- Выдача: diff (track-changes style), копирование только изменений или полного CV, **скачивание PDF**.

### 4.4 Cover Letter
- Длина: **150–250 слов** (макс. 300).
- Стиль: спокойный, взрослый, конкретный; без HR-клише и шаблонных фраз.
- Обязательно: кто/зачем в первых двух предложениях, отсылка к JD, доказательства, спокойное закрытие.
- Ввод опционально: «Что подчеркнуть?» — если пусто, генерация по канону; если есть — канон + пожелания.
- Выдача: **Subject + тело письма**, копирование, **скачивание PDF**.

### 4.5 Pipeline и трекинг
- **Статусы:** New → In Progress → Applied → Waiting → Response / No Response → Interview / Closed.
- Кто ставит: New/Waiting/No Response — система; остальное — пользователь.
- **Тайминги и напоминания:**
  - Stale New: New + 3 дня.
  - Follow-up 1: Applied + 4 дня.
  - Follow-up 2: FU1 + 7 дней.
  - No Response: Applied + 30 дней.
- Хранение: **Google Sheets** как единственная БД (с кэшем в приложении).

### 4.6 UI (веб)
- **Экраны:** Pipeline (таблица) → Карточка вакансии → CV (tailor + аппрув) → Letter (generate + аппрув) → Готово (запись в pipeline).
- **Dashboard:** сводка (всего / новые / подано / ответы), простая воронка, недавняя активность.
- Письмо в таблице: truncate ~50 символов, по клику — popup с полным текстом.
- Один экран = одно решение (минимум когнитивной нагрузки).

### 4.7 Интеграции MVP
- **Google Sheets:** чтение/запись pipeline, один лист (например 21 колонка).
- **Claude API:** скоринг, CV tailor, letter generation.
- **Telegram-бот (текущая версия):**
  - Пишет в тот же Google Sheets (Pipeline).
  - **Без скоринга** и без расхода токенов Claude.
  - При ссылке добавляет карточку сразу; LinkedIn помечает комментом «вставь JD вручную».
  - Парсинг не-LinkedIn URL используется только как проверка доступности текста.
  - Напоминания: в боте работают по датам Follow‑up 1/2 (не по touchpoints).

## 5. Что не входит в MVP (отложено)

| Функция | Причина отложения |
|---------|-------------------|
| BUNDLE (всё сразу без пошаговых аппрувов) | MVP — пошаговый flow с аппрувами |
| A/B варианты писем | Не нужно для первой версии |
| Application form answers (вопрос–ответ) | Позже |
| Выбор BASE_CV (US/INTL) | Одно каноническое резюме |
| Reconnect (переотклик через 90 дней) | Ручное решение в MVP |
| Мультипользователь / SaaS | Один пользователь на инстанс |
| Парсинг LinkedIn через API/scraping | Уточняется отдельно |

## 6. Отличия от типичных продуктов (дифференциаторы)

| Аспект | Joe v2 MVP | Типичные «резюме/письмо» сервисы |
|--------|------------|-----------------------------------|
| **Роль продукта** | Execution agent: только подготовка откликов | Часто: советы, шаблоны, «улучшение» текста без привязки к канону |
| **Честность** | No-Invention зашито в ядро; CANON CHECK; тест «сказал бы на интервью» | Часто допускают «улучшение» формулировок до границы выдумки |
| **Один канон** | Один canonical resume → много адаптаций (акценты, не переписывание) | Часто: каждый раз «новое» резюме или общие шаблоны |
| **Аппрув** | Явный аппрув на каждом шаге (CV, letter) перед использованием | Часто: сгенерировал и отправил без пошагового контроля |
| **Pipeline** | Встроенный трекинг: статусы, напоминания, дубликаты, дашборд | Обычно отдельный ATS/таблица или вручную |
| **Письмо** | Правила в модуле (длина, тон, структура, banned phrases); plug-in заменяемый | Фиксированные шаблоны или свободная генерация без правил |
| **Флаги** | Визы, гражданство, exp_gap, junior_role, strong_mismatch — явные действия (блок/предупреждение) | Редко формализованы в продукте |
| **Данные** | CLIENT_SPACE как единственный источник правды; обновление через snapshot | Разрозненные вводы, история чата, «память» модели |

## 8. Архитектура и реализация (кратко, понятным языком)

### 8.1 Общая схема
- **Frontend (SPA)** → вызывает **Backend API** → записывает/читает **Google Sheets**.
- **AI‑модули** вызываются только из backend.
- **Telegram‑бот** пишет в ту же таблицу (Pipeline), чтобы всё было в одном месте.

### 8.2 Backend (FastAPI)
**Роуты API:**
- `GET/POST /api/jobs` — чтение/добавление вакансий в Pipeline
- `PATCH /api/jobs/{row}` — обновление полей
- `POST /api/jobs/refresh` — сброс кэша
- `POST /api/scoring/evaluate` — оценка соответствия (JD текст/URL)
- `POST /api/cv/tailor` + `POST /api/cv/pdf` — CV
- `POST /api/letter/generate` + `POST /api/letter/pdf` — письмо
- `GET /api/pipeline/stats` — статистика для Dashboard

**Сервисы внутри backend:**
- `sheets` — Google Sheets (Pipeline + Events)
- `parser` — парсинг JD (Jina Reader → BS4 fallback)
- `canon` — канонические данные (CLIENT_SPACE или env fallback)
- `claude` — вызовы Claude API
- `pdf` — сборка PDF (WeasyPrint)
- `contact_parser` — извлечение контакта из JD

### 8.3 Frontend (React)
**Экраны:**
- Pipeline (таблица, фильтры, статусы)
- JobCard (детали, события, подготовка)
- CVScreen / LetterScreen (генерация + апрув)
- Dashboard (сводка + воронка)

### 8.4 Telegram‑бот (текущая версия)
- Polling‑бот на Python.
- Принимает URL или текст, **не оценивает**, сразу создаёт запись в Pipeline.
- LinkedIn → пометка, что JD нужно вставить вручную.
- Напоминания сохраняются (scheduler).
 - Напоминания основаны на колонках Follow‑up 1/2 (не на touchpoints).

### 8.5 Репозиторий и ветки
- **Git repo:** https://github.com/Ksen-Kas/joe_services  
- **MVP‑стек (web):** ветка `mvp-service`
- **Legacy бот:** `main` (Telegram)

## 9. Модульность и расширяемость

- **Letter Module** — подключаемый: замена папки `modules/letter` и правил (letter_rules, examples, banned_phrases) не ломает CV и скоринг.
- **Интерфейс письма:** `generate_letter(jd, canon, notes) → letter`.
- **CLIENT_SPACE** — структура профиля клиента (canonical, strategy, process, snapshots); смена клиента = смена Space, логика Joe та же.

## 10. Критерии готовности задачи (Done)

- Создан хотя бы один артефакт (CV и/или письмо).
- Для CV присутствует CANON CHECK.
- Нет выдуманных фактов.
- Всё готово к вставке/отправке (paste-ready, PDF при необходимости).
